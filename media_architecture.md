# Media Architecture — Shiksha Classes
### Cloudinary Integration Design · Production-Grade

---

## 1. Cloudinary Account Structure

```
Cloud Name:  shiksha-classes
Account:     Single cloud (one Vercel API app has server-side credentials)
```

### Access Model
| Credential | Accessible By | Purpose |
|---|---|---|
| `CLOUDINARY_API_SECRET` | API server only (env var) | Sign upload payloads, destroy assets |
| `CLOUDINARY_API_KEY` | API server + Admin client | Identify the account in uploads |
| `CLOUDINARY_CLOUD_NAME` | Public (frontend + admin) | Construct delivery URLs |

The **API secret never reaches the browser**. All uploads from the Admin portal use a **pre-signed payload** generated by the API.

## 2. Folder Strategy

Every asset is stored under a top-level `shiksha/` namespace. Sub-folders are organised by **asset type**, not by page, which allows assets to be reused across pages and simplifies Cloudinary DAM navigation.

```
shiksha/
├── homepage/
│   ├── hero/               ← Hero background images
│   └── og/                 ← Open Graph / social share images
│
├── results/                ← Student topper photos
│
├── gallery/
│   ├── classrooms/
│   ├── events/
│   ├── results-celebrations/
│   ├── campus-life/
│   ├── faculty/
│   └── awards/
│
├── programs/
│   ├── jee/                ← Program hero + thumbnail images
│   ├── neet/
│   ├── mht-cet/
│   └── previse-foundation/
│
├── blogs/
│   └── covers/             ← Blog post cover images
│
├── resources/
│   ├── thumbnails/         ← Resource card thumbnails
│   └── files/              ← PDF uploads (raw resource type)
│
├── resumes/                ← Job applicant resumes (raw PDF)
│
└── settings/
    └── og/                 ← Default site OG image
```

### Folder Rules
1. **Never** upload to the root `shiksha/` directly — always a sub-folder.
2. The API server's signature generation allowlists exactly these folders. Requests to any other path are rejected.
3. Sub-folders within `gallery/` match the `category` field in the `gallery` MongoDB collection — keeping DB and storage semantically consistent.

## 3. Naming Conventions

### 3.1 Public IDs (Asset Identifiers)

Cloudinary `public_id` is the permanent identifier stored in MongoDB alongside `secureUrl`. It includes the folder path.

**Format:**
```
shiksha/{subfolder}/{semantic-slug}_{timestamp}
```

**Examples:**
```
shiksha/results/priya-deshmukh-jee-2025_1748800000
shiksha/gallery/events/result-celebration-2025_1748800100
shiksha/blogs/covers/how-to-crack-neet_1748800200
shiksha/resources/files/physics-formula-sheet-jee_1748800300
shiksha/resumes/rahul-sharma-physics-faculty_1748800400
```

**Rules:**
- All lowercase.
- Spaces replaced with hyphens.
- Timestamp (Unix seconds) appended to prevent collisions on re-uploads.
- No special characters beyond hyphen and underscore.
- Max public_id length: 100 characters (Cloudinary limit: 255 — stay well under).

### 3.2 Generated by Client vs Server

The `public_id` is constructed **client-side** in the Admin portal before upload:

```typescript
// Admin portal utility
function generatePublicId(folder: string, semanticName: string): string {
  const slug = semanticName
    .toLowerCase()
    .replace(/[^a-z0-9\s-]/g, '')
    .replace(/\s+/g, '-')
    .slice(0, 60);
  const timestamp = Math.floor(Date.now() / 1000);
  return `${folder}/${slug}_${timestamp}`;
}
```

This `public_id` is included in the signed upload parameters so Cloudinary stores it at the exact path.

## 4. Upload Limits

All limits are enforced at **two layers**: client-side (UX feedback before upload) and Cloudinary's upload preset (server-side enforcement).

| Asset Type | Max File Size | Accepted Formats | Max Dimensions |
|---|---|---|---|
| Result / student photos | 2 MB | JPG, PNG, WebP | 1200×1200 px |
| Gallery images | 5 MB | JPG, PNG, WebP | 4000×3000 px |
| Blog cover images | 3 MB | JPG, PNG, WebP | 2400×1350 px |
| Program hero images | 4 MB | JPG, PNG, WebP | 2400×1600 px |
| Resource thumbnails | 1 MB | JPG, PNG, WebP | 800×600 px |
| Homepage hero image | 5 MB | JPG, PNG, WebP | 2560×1440 px |
| Resource PDFs | 10 MB | PDF only | — |
| Applicant resumes | 5 MB | PDF only | — |

### Enforcing at Cloudinary Level (Upload Presets)
Two upload presets are configured in the Cloudinary dashboard:

```
Preset: shiksha-images
  Max file size: 5242880 (5 MB)
  Allowed formats: jpg, png, webp
  Signed: true
  Overwrite: false

Preset: shiksha-raw
  Max file size: 10485760 (10 MB)
  Allowed formats: pdf
  Resource type: raw
  Signed: true
  Overwrite: false
```

The API's `generateSignature()` applies the correct preset per `folder`:
```typescript
const preset = folder.includes('files') || folder.includes('resumes')
  ? 'shiksha-raw'
  : 'shiksha-images';
```

## 5. Compression Strategy

We use Cloudinary's **auto quality (`q_auto`)** and **auto format (`f_auto`)** on all image delivery URLs. This is applied at the transformation level, not stored — the original is always preserved at full quality.

### 5.1 Delivery URL Transform Parameters

Every image `secureUrl` stored in MongoDB is the **original upload URL**. The frontend and admin portal apply transformations at render time via Cloudinary URL parameters.

```
// Original stored URL:
https://res.cloudinary.com/shiksha-classes/image/upload/shiksha/results/priya-deshmukh_1748800000.jpg

// Delivery URL with compression (frontend renders this):
https://res.cloudinary.com/shiksha-classes/image/upload/q_auto,f_auto/shiksha/results/priya-deshmukh_1748800000.jpg
```

### 5.2 Named Transformations

Named transformations are defined once in Cloudinary Dashboard and referenced by name in URLs. This decouples transformation logic from application code.

| Named Transform | Parameters | Used For |
|---|---|---|
| `t_result_card` | `w_400,h_400,c_fill,g_face,q_auto,f_auto,r_8` | Result card thumbnails (square, face-crop, rounded) |
| `t_result_hero` | `w_800,h_800,c_fill,g_face,q_auto,f_auto` | Enlarged result photo on detail view |
| `t_gallery_grid` | `w_600,h_450,c_fill,q_auto,f_auto` | Gallery grid thumbnails (4:3) |
| `t_gallery_lightbox` | `w_1400,q_auto,f_auto` | Full-size gallery view |
| `t_blog_cover_card` | `w_800,h_450,c_fill,q_auto,f_auto` | Blog listing card cover (16:9) |
| `t_blog_cover_hero` | `w_1200,h_630,c_fill,q_auto,f_auto` | Blog post hero + OG image (1200×630) |
| `t_program_hero` | `w_1920,h_600,c_fill,q_75,f_auto` | Program page hero banner |
| `t_program_thumb` | `w_400,h_250,c_fill,q_auto,f_auto` | Program selection card |
| `t_homepage_hero` | `w_2560,q_60,f_auto` | Homepage hero (large, aggressively compressed) |
| `t_og_image` | `w_1200,h_630,c_fill,q_auto,f_auto` | Generic OG social image |
| `t_resource_thumb` | `w_400,h_280,c_fill,q_auto,f_auto` | Resource card thumbnail |
| `t_favicon` | `w_32,h_32,c_fill` | Favicon (if served from Cloudinary) |

**Usage in frontend:**
```typescript
// Utility function used by all image components
function cloudinaryUrl(
  baseSecureUrl: string,
  transform: string
): string {
  // Insert named transform after /upload/
  return baseSecureUrl.replace('/upload/', `/upload/${transform}/`);
}

// Example:
cloudinaryUrl(result.studentImageSecureUrl, 't_result_card');
// → https://res.cloudinary.com/shiksha-classes/image/upload/t_result_card/shiksha/results/...
```

### 5.3 Responsive Images Strategy
The frontend uses the `<picture>` element with `srcSet` for critical images (hero, blog covers) to serve correctly sized images per viewport:

```html
<picture>
  <source media="(max-width: 640px)"  srcset="...t_blog_cover_card..." />
  <source media="(max-width: 1200px)" srcset="...t_blog_cover_hero..." />
  <img src="...t_blog_cover_hero..." alt="..." loading="lazy" />
</picture>
```

## 6. Transformation Strategy by Asset Type

### 6.1 Result / Topper Photos
```
Stored:   Original portrait photo (full res)
Card:     t_result_card    → 400×400, face-crop, rounded corners
Hero:     t_result_hero    → 800×800, face-crop
OG:       t_og_image       → 1200×630 for social sharing of topper highlights
```

**Face detection (`g_face`)** ensures the student's face is always centred in the crop, regardless of how the original photo was framed.

### 6.2 Gallery Images
```
Stored:   Original (up to 4000×3000)
Grid:     t_gallery_grid      → 600×450 (lazy loaded, CDN-cached)
Lightbox: t_gallery_lightbox  → 1400px wide (loaded only on user click)
```

Progressive delivery: Grid loads first with `loading="lazy"`. Lightbox image is fetched on demand — not preloaded.

### 6.3 Blog Cover Images
```
Stored:   Original
Card:     t_blog_cover_card   → 800×450 for /blog listing
Hero:     t_blog_cover_hero   → 1200×630 for post page + OG tag
```

OG meta tag for blog posts:
```html
<meta property="og:image" content="{cloudinaryUrl(coverImageSecureUrl, 't_blog_cover_hero')}" />
```

### 6.4 Homepage Hero
```
Stored:    Original (up to 2560×1440)
Mobile:    w_640,q_60,f_auto    — served via srcSet
Tablet:    w_1280,q_60,f_auto
Desktop:   t_homepage_hero      — 2560px, q_60 (aggressively compressed — biggest file)
```

Hero images are the most performance-critical. Quality is capped at 60 to prioritise LCP speed.

### 6.5 PDFs (Resources & Resumes)
```
Resource type: raw
No transformation applied — PDFs are served as-is via secure URL
Delivery: Cloudinary serves with appropriate Content-Type: application/pdf header
Access:   All PDFs are public (no access control) — no signed delivery URLs needed for resources
Resumes:  Access-controlled — stored in shiksha/resumes/ — admin fetches via authenticated API endpoint that returns the Cloudinary URL
```

## 7. Cleanup Policy

Stale and orphaned assets on Cloudinary incur storage costs and should be actively managed.

### 7.1 Delete-on-Replace Policy

Whenever an asset is **replaced** in the CMS (e.g., admin uploads a new topper photo), the old asset is destroyed immediately:

```typescript
// In the admin API route handler — before saving new publicId to DB:
if (existing.studentImagePublicId) {
  await MediaService.destroy(existing.studentImagePublicId, 'image');
}
// Then save new publicId to MongoDB
```

This applies to:
| Event | Assets Destroyed |
|---|---|
| Result photo replaced | Old `studentImagePublicId` |
| Blog cover replaced | Old `coverImagePublicId` |
| Resource PDF replaced | Old `filePublicId` |
| Resource thumbnail replaced | Old `thumbnailPublicId` |
| Gallery image deleted | `imagePublicId` |
| Program hero replaced | Old `heroImagePublicId` |
| Homepage hero replaced | Old `backgroundImagePublicId` |
| Career application deleted | `resumePublicId` |

### 7.2 Delete-on-Document-Delete Policy

When a MongoDB document is deleted, its Cloudinary assets are destroyed:

| Document Deleted | Cloudinary Assets Destroyed |
|---|---|
| `results` record | `studentImagePublicId` |
| `blogs` record | `coverImagePublicId`, `ogImagePublicId` (if separate) |
| `resources` record | `filePublicId`, `thumbnailPublicId` |
| `gallery` record | `imagePublicId` |
| `career_applications` record | `resumePublicId` |

### 7.3 Orphan Detection (Monthly Maintenance)

A monthly admin-triggered audit script identifies assets in Cloudinary that have no corresponding MongoDB record:

```typescript
// Scratch script: tools/cloudinary-audit.ts
// Run manually: npx ts-node tools/cloudinary-audit.ts

const cloudinaryAssets = await cloudinary.api.resources({
  type: 'upload',
  prefix: 'shiksha/',
  max_results: 500,
});

const publicIds = cloudinaryAssets.resources.map(r => r.public_id);

// Cross-reference against all publicId fields in MongoDB
// Assets with no DB reference → candidate for deletion (admin reviews first)
```

### 7.4 Soft-Delete Safety Net

Before any Cloudinary `destroy()` call, the API verifies the `publicId` is not referenced by any other MongoDB document. This prevents accidental deletion of shared assets (e.g., if the same image URL was stored in two places).

```typescript
async function safeDestroy(publicId: string, resourceType: 'image' | 'raw') {
  // Check no other document still references this publicId
  const refCount = await countReferencesInDB(publicId);
  if (refCount > 0) {
    logger.warn(`Skipping Cloudinary destroy — publicId still referenced: ${publicId}`);
    return;
  }
  await cloudinary.uploader.destroy(publicId, { resource_type: resourceType });
}
```

## 8. Signed Upload Workflow (Admin Portal)

The admin portal **never** sends file bytes through the Vercel API. Files are uploaded directly from the browser to Cloudinary using a time-limited signed payload. This bypasses Vercel Serverless's 4.5 MB request body limit.

```
┌─────────────┐     1. Request signature       ┌─────────────────┐
│ Admin Portal│ ──────────────────────────────► │  Vercel API     │
│ (Browser)   │     POST /api/admin/            │  (Serverless)   │
│             │     media/signature             │                 │
│             │ ◄────────────────────────────── │  Signs with     │
│             │     { signature, timestamp,     │  API Secret     │
│             │       apiKey, cloudName,        │                 │
│             │       folder }                  │                 │
└─────────────┘                                 └─────────────────┘
      │
      │ 2. Upload file directly to Cloudinary
      │    POST https://api.cloudinary.com/v1_1/
      │         shiksha-classes/image/upload
      │    Body: { file, signature, timestamp, apiKey, folder, public_id }
      ▼
┌──────────────┐
│  Cloudinary  │ ──► Returns: { secure_url, public_id, format, bytes, width, height }
└──────────────┘
      │
      │ 3. Save metadata to MongoDB via API
      │    POST /api/admin/results
      │    Body: { ...formData, studentImageSecureUrl, studentImagePublicId }
      ▼
┌─────────────────┐
│  Vercel API     │ ──► Saves to MongoDB Atlas
└─────────────────┘
```

### Signature Expiry
Cloudinary signatures expire **1 hour** after the `timestamp`. The upload must complete within this window.

### Upload Preset Enforcement
The signature locks down:
- Allowed `folder` (from allowlist)
- `resource_type` (`image` or `raw`)
- Upload preset (`shiksha-images` or `shiksha-raw`)

An attacker with an intercepted signature **cannot** upload to a different folder or bypass file type restrictions — these are baked into the HMAC signature.

## 9. Admin Workflow: Media Upload Per Module

### 9.1 Results Manager — Student Photo Upload
```
1. Admin clicks "Upload Photo" in result form
2. File picker opens → accepts: .jpg, .jpeg, .png, .webp
3. Client validates: file size ≤ 2 MB (shows error if exceeded)
4. Client renders immediate preview using FileReader (before upload)
5. Admin fills other fields while photo preview shows
6. Admin clicks "Save Result":
   a. Client calls POST /api/admin/media/signature { folder: 'shiksha/results', resourceType: 'image' }
   b. Client constructs public_id: generatePublicId('shiksha/results', studentName + '-' + program + '-' + year)
   c. Client uploads to Cloudinary with signature
   d. Cloudinary returns { secure_url, public_id }
   e. Client includes secure_url + public_id in POST /api/admin/results body
   f. Server saves to MongoDB
7. Success → result card appears in list
```

### 9.2 Gallery Manager — Single Image Upload
```
1. Admin opens Upload Modal
2. Drop zone or file picker → accepts: .jpg, .jpeg, .png, .webp, max 5 MB
3. Image preview rendered immediately (client-side)
4. Admin fills: Alt Text (required), Caption, Category, Event Name, Date
5. Admin clicks "Upload & Save":
   a. Client builds folder path: 'shiksha/gallery/' + categorySlug
   b. Requests signature for that folder
   c. Uploads to Cloudinary
   d. Saves record to MongoDB with secureUrl + publicId
6. Gallery grid refreshes with new image
```

### 9.3 Gallery Manager — Bulk Upload
```
1. Admin selects multiple files (up to 20)
2. File queue appears — each row shows: filename, preview, size, status indicator
3. Admin selects shared Category and optional Event Name
4. Admin can optionally set per-file Caption and Alt Text inline
5. Admin clicks "Upload All":
   → Sequential upload: one file at a time (avoids Cloudinary concurrent request limits)
   → Each file: request signature → upload → save MongoDB record → update status indicator
6. Final summary: "18 uploaded, 2 failed"
   → Failed files remain in queue — admin can retry or skip
```

### 9.4 Blog Manager — Cover Image Upload
```
1. Admin is in blog editor
2. "Cover Image" section shows: empty drop zone or existing preview
3. Admin drops/selects file → immediate preview
4. Upload happens automatically on file select (eager upload):
   a. Request signature for 'shiksha/blogs/covers'
   b. Upload to Cloudinary
   c. Store returned secure_url + public_id in editor state
   d. Small "✓ Uploaded" badge appears on preview
5. On "Save Draft" or "Publish" → secureUrl included in blog payload
```

### 9.5 Resources Manager — PDF Upload
```
1. Admin creates/edits resource
2. "Attach File" section → file picker, accepts: .pdf only, max 10 MB
3. Progress bar shown during upload (PDFs are larger — progress feedback is important)
4. Upload:
   a. Request signature for 'shiksha/resources/files' with resourceType: 'raw'
   b. Upload using Cloudinary raw endpoint
   c. Cloudinary returns { secure_url, public_id, bytes, format }
5. File info displayed: filename, size in KB/MB, "✓ Uploaded"
6. Saved with resource record
```

### 9.6 Careers — Resume Upload (Public, Not Admin)
```
1. Applicant fills job application form on public /careers/[slug] page
2. "Upload Resume" field → accepts: .pdf only, max 5 MB
3. Unlike admin uploads, the public endpoint:
   a. Frontend calls POST /api/careers/apply/resume-signature (public, rate-limited endpoint)
   b. Server generates signature for shiksha/resumes/ folder ONLY
   c. No other folder is signable from this public endpoint
   d. Cloudinary enforces PDF-only via upload preset
4. Resume uploaded directly to Cloudinary from browser
5. secure_url + public_id sent with application payload to POST /api/careers/apply
```

## 10. CDN Delivery & Caching

Cloudinary's CDN (backed by Fastly) automatically caches transformed images at edge nodes globally.

### Cache Behaviour
| Asset | CDN Cache TTL | Invalidation |
|---|---|---|
| Named transform URLs | 1 year (immutable) | Changing the public_id effectively busts cache |
| Original upload URLs | 1 year | Same — new public_id on replace |
| PDF raw files | 1 hour | Manual purge if needed |

Since we **never modify** assets in-place (always upload a new asset, store the new publicId, destroy the old one), URL-based cache invalidation is automatic — old URLs are abandoned.

### HTTPS Enforcement
All Cloudinary URLs use `https://` (`secure: true` set in SDK config). HTTP is never used.

## 12. Summary: Key Design Decisions

| Decision | Rationale |
|---|---|
| Signed client-side uploads | Bypasses Vercel's 4.5 MB serverless request limit |
| Store `publicId` alongside `secureUrl` | Enables server-side Cloudinary asset deletion |
| Named transformations | Decouples transform logic from app code; one place to change |
| `q_auto,f_auto` on all deliveries | Optimal quality + smallest file size per browser (WebP in Chrome, JPEG elsewhere) |
| `g_face` on student photos | Ensures face-centred crop regardless of original framing |
| Folder-per-category in gallery | DB `category` field and Cloudinary folder are always in sync |
| Original preserved at full res | Future-proof — new named transforms can be applied without re-uploading |
| Delete-on-replace policy | Prevents Cloudinary storage bloat; keeps costs predictable |
